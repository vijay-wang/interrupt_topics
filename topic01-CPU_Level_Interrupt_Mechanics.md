## 1. Position in the Subsystem

This topic covers the segment of the spine between the CPU pin and `generic_handle_*`:

```
controller delivers ──► CPU vectors to a stub ──► save state, switch stack,
                        (Topic 02)                generic-entry bookkeeping
                                                  ──► call into kernel/irq/ (Topic 05)
```

The crucial architectural difference you will learn here:

- **On x86, the *vector* already identifies the interrupt.** The APIC delivered a specific vector (0–255); the per-CPU `vector_irq[vector]` array maps it straight to an `irq_desc`. The hardware did the demux.
- **On arm64, one root handler reads the controller.** The CPU only knows "an IRQ happened"; a single registered `handle_arch_irq` (e.g. the GIC driver) reads the controller register to discover *which* hwirq fired, then maps it.

That contrast explains a lot of later design.

---

## 2. What the CPU Does on an Interrupt (hardware)

Independent of OS, on an asynchronous interrupt the core:

1. Finishes or abandons the in-flight instruction to reach a precise boundary.
2. **Masks** further interrupts of equal/lower priority (so the handler isn't immediately re-entered).
3. Saves the minimum architectural state needed to return (at least PC and processor status).
4. Possibly switches privilege level (user→kernel) and stack.
5. Jumps to a hardware-defined handler address derived from a *vector* (x86) or *vector table slot* (arm64).

Everything after step 5 is software — and that software is the subject of the rest of this topic.

## 3. x86_64 Entry

### 3.1 The IDT and vectors
The **Interrupt Descriptor Table** has 256 gates. The 8-bit *vector* the Local APIC delivers indexes it. Layout of the vector space:

- **0–31**: CPU exceptions (#PF, #GP, #DF, NMI=2, …) — not device IRQs.
- **32–255**: external device vectors, IPIs, and special system vectors. Device vectors are a **scarce per-CPU resource** handed out by the matrix allocator (Topic 13); the upper end holds fixed system vectors (reschedule, call-function, APIC error/spurious, thermal, posted-MSI, …).

Gate types matter: an **interrupt gate** clears IF on entry (interrupts masked); a **trap gate** does not. Device IRQs use interrupt gates.

### 3.2 The `idtentry` machinery
Entry stubs are generated by macros in `arch/x86/include/asm/idtentry.h`: `DECLARE/DEFINE_IDTENTRY`, `DEFINE_IDTENTRY_IRQ` (device IRQs), `DEFINE_IDTENTRY_SYSVEC` (system vectors like IPIs), `DEFINE_IDTENTRY_IRQENTRY`, plus IST variants. They wrap the C handler in the asm prologue/epilogue and the generic enter/exit calls.

The canonical device path (`arch/x86/kernel/irq.c`):

```c
DEFINE_IDTENTRY_IRQ(common_interrupt)        /* the gate for all ordinary device vectors */
{
    struct pt_regs *old_regs = set_irq_regs(regs);
    RCU_LOCKDEP_WARN(!rcu_is_watching(), "IRQ failed to wake up RCU");
    if (unlikely(!call_irq_handler(vector, regs)))   /* vector → irq_desc → handle_irq */
        apic_eoi();
    set_irq_regs(old_regs);
}

static __always_inline bool call_irq_handler(int vector, struct pt_regs *regs)
{
    struct irq_desc *desc = __this_cpu_read(vector_irq[vector]);   /* the demux! */
    if (likely(!IS_ERR_OR_NULL(desc))) {
        handle_irq(desc, regs);                  /* switches to IRQ stack, runs desc->handle_irq */
        return true;
    }
    /* else: stale/unassigned vector → re-check under vector_lock, else EOI & drop */
}
```

So on x86: **vector → `vector_irq[vector]` → `irq_desc` → flow handler**. The `DEFINE_IDTENTRY_IRQ` wrapper already ran the generic enter (§5) before the body; `handle_irq()` performs the IRQ-stack switch.

### 3.3 IRQ stacks and IST
Ordinary device IRQs run on a per-CPU **hardirq stack**, switched to by `run_irq_on_irqstack_cond()` (`arch/x86/include/asm/irq_stack.h`) inside `handle_irq()` — but only if not already on it (nested IRQs reuse the stack).

Some *exceptions* need a guaranteed-good stack regardless of the current one: the **IST** (Interrupt Stack Table) gives NMI, #DF (double fault), #DB, #MC, and #VC dedicated stacks (`enum`, `arch/x86/include/asm/cpu_entry_area.h`: `ESTACK_DF/NMI/DB/MCE/VC`). IST is why an NMI can fire safely even mid-stack-overflow. Device IRQs do **not** use IST; only these exceptions do.

## 4. arm64 Entry

### 4.1 The exception vector table
`VBAR_EL1` points at `vectors` (`arch/arm64/kernel/entry.S`), a 16-entry table = 4 exception sources × 4 origins:

```
SYM_CODE_START(vectors)
    kernel_ventry 1, t, 64, sync   // EL1t synchronous
    kernel_ventry 1, t, 64, irq    // EL1t IRQ
    kernel_ventry 1, t, 64, fiq    // EL1t FIQ
    kernel_ventry 1, t, 64, error  // EL1t SError
    kernel_ventry 1, h, 64, sync   // EL1h ...
    kernel_ventry 1, h, 64, irq
    ...                            // then EL0 64-bit, then EL0 32-bit
```

The four origins are: same-EL with SP_EL0 (`t`), same-EL with SP_ELx (`h`), lower-EL AArch64 (EL0 64-bit), lower-EL AArch32. Each provides **sync / IRQ / FIQ / SError** entries. So "an IRQ taken from kernel mode" lands in the `el1h_64_irq` slot; "from user mode" in `el0t_64_irq`.

`ESR_EL1` (Exception Syndrome) classifies *synchronous* exceptions; `FAR_EL1` holds the faulting address. IRQs are asynchronous so they don't use ESR — they go straight to the controller.

### 4.2 From the vector to the C handler
The asm slot calls into C (`arch/arm64/kernel/entry-common.c`):

```c
asmlinkage void noinstr el1h_64_irq_handler(struct pt_regs *regs)
{ el1_interrupt(regs, handle_arch_irq); }

static void noinstr el1_interrupt(struct pt_regs *regs, void (*handler)(struct pt_regs *))
{
    write_sysreg(DAIF_PROCCTX_NOIRQ, daif);          /* keep IRQs masked while we run */
    if (IS_ENABLED(CONFIG_ARM64_PSEUDO_NMI) && regs_irqs_disabled(regs))
        __el1_pnmi(regs, handler);                   /* pseudo-NMI path */
    else
        __el1_irq(regs, handler);
}

static __always_inline void __el1_irq(struct pt_regs *regs, void (*handler)(struct pt_regs *))
{
    irqentry_state_t state = arm64_enter_from_kernel_mode(regs);   /* generic enter (§5) */
    irq_enter_rcu();
    do_interrupt_handler(regs, handler);             /* → call_on_irq_stack → handler(regs) */
    irq_exit_rcu();
    arm64_exit_to_kernel_mode(regs, state);          /* generic exit (§5) */
}
```

`handle_arch_irq` is a function pointer the **root interrupt controller** registered at init (`set_handle_irq()` — Topic 17). For a GICv3 system it is `gic_handle_irq`, which reads `ICC_IAR1_EL1` to learn the hwirq, then calls `generic_handle_domain_irq(domain, hwirq)`. That is the demux x86 did in hardware.

`do_interrupt_handler` switches to the per-CPU IRQ stack via `call_on_irq_stack()` if currently on the thread stack:

```c
if (on_thread_stack()) call_on_irq_stack(regs, handler);
else                   handler(regs);
```

arm64 also supports **pseudo-NMI** (`CONFIG_ARM64_PSEUDO_NMI`): using GIC priorities, a high-priority IRQ can act like an NMI even though the architecture historically lacked a true maskable-separate NMI line (Topic 12).

## 5. The Generic Entry/Exit Layer (shared by both arches)

Both arches funnel through `include/linux/irq-entry-common.h`, which centralizes the bookkeeping that *must* happen on every kernel entry so the rest of the kernel's invariants hold:

- **`irqentry_enter(regs)` → `irqentry_state_t`** and **`irqentry_exit(regs, state)`** for ordinary interrupts from kernel mode; `irqentry_enter_from_user_mode` / `irqentry_exit_to_user_mode` for the user-mode case (x86 uses these directly; arm64 wraps them as `arm64_enter/exit_from_kernel_mode`).
- **What they do:** tell RCU the CPU is no longer idle/quiescent (`rcu_irq_enter` semantics), update context tracking (for `nohz_full`/`CONTEXT_TRACKING`), maintain lockdep's hardirq on/off state, and on the *exit* path decide whether to preempt (`need_resched` when returning to kernel) or take the return-to-user work loop (signals, etc.).
- **`irqentry_nmi_enter/exit`** is the NMI-safe variant (no RCU/lockdep operations that could themselves deadlock); arm64's `__el1_pnmi` uses it.

This is why the comment in `common_interrupt` can `RCU_LOCKDEP_WARN(!rcu_is_watching())`: by the time the C body runs, the generic enter has guaranteed RCU is watching. **Topic 10** revisits the lockdep/RCU/`preempt_count` side from the concurrency angle; here the point is simply: *every interrupt passes through one common gate that keeps RCU, context-tracking, lockdep, and preemption honest.*

---

## 6. Return From Interrupt

On the exit path (the generic `irqentry_exit` / arm64 `arm64_exit_to_*`):

- **Returning to kernel:** if preemption is enabled and `need_resched` is set, call the preemption path (`preempt_schedule_irq`). Restore the saved state and `iret`/`eret`.
- **Returning to user:** run the return-to-user work loop — pending signals, `TIF_NOTIFY_RESUME`, rescheduling — before restoring user state.
- Pending **softirqs** raised during the handler are run earlier, in `irq_exit_rcu()` (the hardirq tail) — that is the seam where Topic 08 attaches.

## 7. Key Design Rationale

- **Why a separate IRQ stack?** Kernel thread stacks are small (often 16 KiB). A deeply nested call chain that then takes an interrupt could overflow. A dedicated per-CPU IRQ stack bounds the worst case; nesting reuses it rather than stacking again.
- **Why IST (x86) / always-valid stacks?** Some exceptions (NMI, double fault) can occur when the current stack is *itself* corrupt or mid-switch. A hardware-guaranteed fresh stack is the only safe option.
- **Why a generic entry layer at all?** RCU, context tracking, lockdep, and preemption each need exact "we just entered/left the kernel" hooks. Before the generic layer these were duplicated and subtly wrong per-arch. Centralizing them (`irq-entry-common.h`, `kernel/entry/`) made the invariants uniform — directly serving R1's "cover the *why*, not just the *what*".
- **Why x86 demuxes in hardware but arm64 in software?** APIC delivers a vector that already encodes the source; the GIC delivers a generic IRQ exception and you must read `IAR` to learn the source. Neither is "better" — it shapes the controller drivers (Topic 02) and the affinity/vector model (Topic 13).

## 8. Observability & Pitfalls

### Observability
- `/proc/interrupts` — the rightmost text shows the arch/controller side; the per-CPU columns show where vectors land (x86) or where the root handler dispatched (arm64).
- x86: `vector_irq` is per-CPU; a vector with no `irq_desc` is `VECTOR_UNUSED/SHUTDOWN/RETRIGGERED` (see `arch/x86/include/asm/hw_irq.h`).
- Tracing the entry: `function_graph` on `common_interrupt` (x86) or `el1h_64_irq_handler`/`gic_handle_irq` (arm64).

### Pitfalls
- **"IRQ failed to wake up RCU" splat** — something entered an IRQ without the generic enter; almost always a broken custom entry path. The `RCU_LOCKDEP_WARN` in `common_interrupt` is exactly this guard.
- **Confusing exceptions with IRQs** — `#PF` (CR2/page fault) is *synchronous* and never touches `irq_desc`; do not look for it in `/proc/interrupts`.
- **Stack overflow showing up as a double fault (x86) or SError (arm64)** — because the IRQ/exception stacks are where the spill lands; read Topic 16 for diagnosis.
- **Assuming IRQs are off inside a handler** — true initially (interrupt gate / `DAIF_PROCCTX_NOIRQ`), but flow handlers and threaded handlers may re-enable; never rely on implicit masking for data races (Topic 10).
